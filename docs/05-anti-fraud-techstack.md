# AI Anti-Fraud 审计助手 — 完整产品方案 + 面试视角审查

> 产品定位：帮 SME「看清每一笔账」的智能审计空间
> 目标用户：Finance Controller / CFO（团队小、预算有限、无内审部门）
> 约束：4 周 MVP 上线，持续迭代
> 架构演进原则：先单体 MVP → 有真实痛点再拆 Agent → 数据够了再上 ML

## 一、迭代路线全景图（每轮都讲清 WHY / VALUE / BAD CASE / 评测）

---

### V1 — MVP（Week 1-4）：单 Agent，一条管线跑通

#### 架构

```
用户上传凭证 → [单体 FastAPI 服务]
                  ├── Mistral OCR（提取文字）
                  ├── Claude Sonnet（结构化字段 + 审计推理）
                  ├── 规则引擎（10 条硬编码规则）
                  └── 飞书通知（推送结果）
```

**没有 Agent 编排，就是一条顺序管线。**

#### 为什么这么做

- 4 周内必须上线，任何多 Agent 编排都会引入调试复杂度
- 核心假设只有一个：**全量扫描比 5% 抽样更能帮 CFO 发现问题**——这个假设不需要复杂架构来验证
- 单体服务 = 一个人能维护、一个 Docker 能跑、一条日志能 debug

#### 带来的价值

- CFO 第一次看到 100% 凭证都被扫过，每笔都有风险标记——这是「wow moment」
- 飞书群里实时推送高风险项——从「事后审计」变成「实时监控」
- 量化价值：假设 1000 笔/月，传统抽样 50 笔，现在 1000 笔全扫 → **覆盖率从 5% → 100%**

#### Bad Cases（V1 已知会出的问题）

| Bad Case | 原因 | 影响 | V1 应对 | 彻底解决 |
|---|---|---|---|---|
| OCR 提取错误 → 金额/日期识别错 → 误报 | 凭证照片模糊、手写体、非标格式 | 用户信任度下降 | 人工标注入口 + 置信度展示 | V1.1 多 OCR 引擎投票 |
| Claude 审计推理幻觉 → 编造不存在的风险 | LLM 固有问题 | 虚假警报泛滥 | 规则引擎结果优先，LLM 仅作辅助判断 | V2 评测集 + 人工审核校准 |
| 规则过于粗暴 → 整数金额全标红 | 硬编码规则无行业上下文 | 误报率高（预估 30-40%） | 允许用户「标记误报」→ 收集数据 | V1.2 统计基线替代硬阈值 |
| 同一笔交易 OCR + 规则 + LLM 结论矛盾 | 三层逻辑独立，无仲裁 | 用户困惑 | 以规则引擎为主，LLM 为辅 | V2 引入 Orchestrator Agent |
| 大批量上传（500+ 张）超时 | 顺序处理，无异步队列 | 页面卡死 | FastAPI BackgroundTasks | V1.1 Redis Queue |

#### 评测集（V1 必须建）

**这是最重要的基础设施，没有评测集的 AI 产品就是在裸奔。**

```
eval_dataset/
├── golden_set/          # 100 张人工标注的凭证（必须）
│   ├── invoices/        # 30 张发票（含 10 张已知异常）
│   ├── receipts/        # 30 张收据（含 10 张已知异常）
│   ├── bank_statements/ # 20 份流水（含 5 份已知异常）
│   └── labels.json      # 人工标注: 每张的正确字段 + 应触发的规则 + 正确风险等级
├── adversarial/         # 20 张刻意构造的边界 case
│   ├── blurry_scan.jpg
│   ├── handwritten_receipt.jpg
│   ├── duplicate_invoice_pair/
│   ├── split_billing_set/      # 拆单场景
│   └── fake_tax_number.pdf
└── regression/          # 每次改 prompt/规则后必须回归
    └── previous_failures/  # 曾经判断错误的 case
```

**评测指标（V1）：**

| 指标 | 目标 | 计算方式 |
|---|---|---|
| OCR 字段准确率 | ≥ 95% | 提取字段 vs golden label 精确匹配 |
| 规则触发 Precision | ≥ 70% | 标红的里面真有问题的比例 |
| 规则触发 Recall | ≥ 90% | 真有问题的被标出来的比例 |
| LLM 审计一致性 | ≥ 80% | 同一张凭证跑 3 次，结论一致的比例 |
| 端到端延迟 | ≤ 30s/张 | 上传到结果展示 |

**为什么 Recall > Precision？** 审计场景漏检的代价远大于误报。CFO 可以容忍多看几个假警报，但不能容忍真正的欺诈被放过。

### V1.1 — 反馈闭环（上线后 2-4 周）：引入 Human-in-the-Loop

#### 架构变化

```
V1 管线不变，新增：
  ├── 用户反馈收集（标记误报 / 漏报 / 纠正字段）
  ├── 反馈数据 → 规则参数自动调优
  └── Prompt 版本管理（每次改 prompt 存版本 + 跑回归）
```

#### 为什么这么做

- V1 上线后一定会收到大量「这个标红了但没问题」的反馈——**误报是用户流失的第一原因**
- 没有反馈闭环的 AI 产品 = 开环控制 = 永远不会变好
- 这一步不需要 ML，只需要把规则阈值从硬编码变成**可按客户/行业配置**

#### 带来的价值

- 误报率从 30-40% 降到 15-20%（基于反馈调参）
- 用户感受到「系统在学习」→ 信任度上升
- 积累标注数据 → 为 V2 的 ML 模型打基础

#### Bad Cases

| Bad Case | 应对 |
|---|---|
| 用户标记误报但实际是真风险（CFO 自己想掩盖） | 审计日志不可删改；管理员 vs 审计员权限分离 |
| 不同客户反馈矛盾（A 觉得整数金额不算异常，B 觉得算） | 按客户/行业分组的规则配置，不全局调 |
| Prompt 改了一版，旧 case 修好了，新 case 又坏了 | **Prompt 版本管理 + 每次改完跑回归测试集** |

#### 评测集更新

- 把所有用户反馈的误报/漏报加入 `regression/` 目录
- 建立 CI pipeline：每次 prompt 或规则变更 → 自动跑 golden_set → Precision/Recall 不能回退

### V2 — 多 Agent 架构（上线后 2-3 月）：拆分职责，引入 Orchestrator

#### 为什么这时候才引入多 Agent

V1/V1.1 暴露的真实问题会逼你拆 Agent：
- OCR 提取和审计推理耦合在一条 prompt 里 → 改一个坏另一个
- 不同类型凭证（发票 vs 银行流水 vs 收据）需要不同处理逻辑 → 单 prompt 越来越臃肿
- 需要「交叉验证」→ 一个 Agent 的结论需要另一个 Agent 来 challenge

#### 架构

```
                    ┌─────────────────┐
                    │  Orchestrator   │  ← 调度中心，决定流程
                    │  Agent          │
                    └────┬───┬───┬────┘
                         │   │   │
              ┌──────────┘   │   └──────────┐
              ▼              ▼              ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │ Extraction │  │   Audit    │  │  Reviewer  │
     │   Agent    │  │   Agent    │  │   Agent    │
     │            │  │            │  │            │
     │ OCR+结构化  │  │ 规则+推理   │  │ 交叉验证    │
     │ 按凭证类型  │  │ 风险评估    │  │ 质疑+确认   │
     │ 分 prompt  │  │ 统计基线    │  │ 最终裁定    │
     └────────────┘  └────────────┘  └────────────┘
```

**三个 Agent 各司其职：**

| Agent | 职责 | 为什么要拆出来 |
|---|---|---|
| **Extraction Agent** | OCR + 字段提取 + 分类 | 发票/收据/流水的提取逻辑完全不同，prompt 需独立优化 |
| **Audit Agent** | 规则引擎 + 统计基线 + LLM 推理 | 审计逻辑随行业/客户不同，需要独立迭代 |
| **Reviewer Agent** | 交叉验证 + 矛盾仲裁 + 最终风险裁定 | V1 中「规则说有问题、LLM 说没问题」的矛盾需要有人裁 |

#### 带来的价值

- 各 Agent 独立迭代：改提取不影响审计，改审计不影响裁定
- Reviewer Agent 降低 LLM 幻觉的影响（双重校验）
- 可以按凭证类型并行处理 → 吞吐量 3-5x 提升

#### Bad Cases

| Bad Case | 原因 | 应对 |
|---|---|---|
| Agent 间通信延迟 → 整体变慢 | 三次 LLM 调用 vs V1 的一次 | Extraction 用 Haiku（快+便宜），Audit 用 Sonnet，Reviewer 仅处理有争议的 |
| Orchestrator 路由错误 → 发票被当收据处理 | 分类模型不够准 | 用户上传时可手动选凭证类型（兜底） |
| 三个 Agent 结论打架 → Reviewer 也拿不准 | 真正的边界 case | 标记为「需人工复核」→ 不强行给结论 |

#### 评测集更新

- 拆分为三层独立评测：Extraction Eval / Audit Eval / Reviewer Eval
- 新增：端到端一致性测试（同一张凭证，V1 和 V2 结论不应有重大偏差）
- 新增：A/B 测试框架（V1 单管线 vs V2 多 Agent，比较 Precision/Recall/延迟/成本）

### V3 — 智能化（上线后 4-6 月）：数据飞轮转起来

#### 架构新增

```
V2 多 Agent 架构 +
  ├── Baseline Agent（动态风险画像）
  │   └── 基于 3 个月历史数据建立员工/供应商行为基线
  ├── Graph Agent（关系网络分析）
  │   └── Neo4j: 供应商-员工-交易三角关系 → 团伙欺诈检测
  └── Learning Agent（模型自训练）
      └── 用累积的标注数据微调异常检测模型
```

#### 为什么这时候才做

- **动态基线**需要 ≥ 3 个月的交易数据才有统计意义——V1/V2 阶段数据量不够
- **图谱分析**需要足够多的供应商和员工节点——10 个客户以上才有网络效应
- **模型微调**需要 ≥ 1000 条人工标注数据——V1.1 的反馈闭环到这时候才积累够

#### 带来的价值

- 从「规则审计」升级到「智能审计」：系统能发现人和规则都没想到的异常
- 数据飞轮：用得越多 → 标注越多 → 模型越准 → 用户越依赖 → 竞争壁垒
- 图谱分析可以抓到规则引擎永远抓不到的「合谋欺诈」

#### Bad Cases

| Bad Case | 影响 | 应对 |
|---|---|---|
| 基线冷启动：新客户没有历史数据 | 前 3 个月无法用动态基线 | 提供行业默认基线（需收集行业数据） |
| 图谱误关联：同名不同供应商 | 虚假关联 → 误报 | 实体消歧（税号+地址+银行账号综合判断） |
| 模型 drift：客户业务变化导致基线失效 | 正常交易被标异常 | 滑动窗口（最近 90 天）+ 异常基线自动重建 |

## 二、评测集——为什么是最重要的基础设施

### AI PM 必须理解的一句话

> **没有评测集的 AI 产品，就像没有单元测试的代码——你永远不知道改了什么会炸什么。**

### 评测集解决的核心问题

```
PM: "我们的 AI 准确率是多少？"
工程师: "感觉还行。"     ← 没有评测集
工程师: "Precision 78%, Recall 92%。上次改 prompt 后
         Precision 涨了 3%，但收据类的 Recall 跌了 2%，
         原因是这 3 个 case..."  ← 有评测集
```

### 评测集建设的阶段性目标

| 阶段 | 规模 | 来源 | 用途 |
|---|---|---|---|
| V1 | 100 张 golden + 20 张 adversarial | 手工构造 + 公开数据集 | 基本功能验证 |
| V1.1 | +200 张用户反馈 | 真实误报/漏报 | Prompt 回归测试 |
| V2 | +500 张多类型 | 客户真实数据（脱敏） | 分 Agent 独立评测 |
| V3 | 2000+ 张 | 持续积累 | 模型微调 + A/B 测试 |

### 必须跟踪的指标演进

| 指标 | V1 目标 | V2 目标 | V3 目标 |
|---|---|---|---|
| Precision（标红=真有问题） | ≥ 70% | ≥ 85% | ≥ 92% |
| Recall（真问题被标出） | ≥ 90% | ≥ 93% | ≥ 96% |
| OCR 字段准确率 | ≥ 95% | ≥ 97% | ≥ 99% |
| 端到端延迟/张 | ≤ 30s | ≤ 15s | ≤ 5s |
| 月成本/千笔 | ~$60 | ~$40 | ~$20 |

## 三、大厂 AI PM 面试官会追问的 15 个问题

### A. 产品定义 & 用户理解

**Q1: "全量扫描"真的是用户要的吗？还是你假设的？**
> CFO 的真实痛点可能不是"扫描覆盖率"，而是"出了事我能交代"（合规兜底）或"我不想月底加班做审计"（效率）。你验证过哪个是真需求吗？
>
> **你的盲点**：你默认"100% 覆盖 > 5% 抽样"是 wow moment，但如果误报太多，100% 覆盖反而 = 100% 噪音。**用户愿意看的不是"全部标记"，而是"帮我找到那 3 笔真正有问题的"。**

**Q2: SME 的 CFO 真的会自己上传凭证吗？**
> 大部分 SME 的凭证在会计手里，CFO 只看报告。你的用户旅程是 CFO 驱动采购 → 会计日常使用 → CFO 看 Dashboard？如果是，你的产品要同时服务两个角色，交互设计完全不同。

**Q3: 定价怎么做？$65-130/月的成本，客户愿意付多少？**
> SME 的财务软件预算通常 ¥500-2000/月。你的 LLM 成本跟交易量线性增长——如果客户量大，你的 margin 会被 API 成本吃掉。有没有想过阶梯定价或 per-voucher 计费？

**Q4: "审计底稿级别"是什么标准？你跟真正的审计师确认过吗？**
> 如果你的产品输出不能被四大认可为审计证据，那"审计底稿级别"就是个市场话术而非真实能力。这个定位会影响你的合规风险。

### B. 技术架构 & AI 能力

**Q5: 你的 AI 审计结论的法律效力是什么？出了事谁担责？**
> 如果 AI 说"这笔没问题"，结果是欺诈，CFO 能拿你的报告作为免责证据吗？如果不能，你的产品本质上是"建议工具"而非"审计工具"——这决定了你的 GTM 话术。
>
> **你的盲点**：整个方案没有提到法律责任声明和产品边界定义。

**Q6: 为什么用 Claude 而不是微调一个专用审计模型？**
> 期望的回答：MVP 阶段 prompt engineering 比微调快 10 倍；等积累 2000+ 标注数据（约 V3 阶段）再考虑微调；Claude 的 structured output 能力是当前最好的，审计需要结构化输出。
>
> 追问：你有没有评估过如果 Claude API 涨价 50% 或 Anthropic 修改 ToS 禁止金融审计场景，你的 Plan B 是什么？

**Q7: 规则引擎和 LLM 的职责边界在哪里？**
> "金额为整数且 > 5000"这种规则是确定性的，但"这笔交易看起来可疑"是概率性的。当规则说"没问题"但 LLM 说"可疑"时，谁赢？你的仲裁逻辑是什么？
>
> **你的盲点**：V1 方案里规则和 LLM 是并列的，但没有定义冲突仲裁策略。

**Q8: 你怎么处理 LLM 的不确定性和不一致性？**
> 同一张发票跑两次 Claude，结论可能不同。审计场景对一致性要求极高。你的方案里没有提到 temperature 设置、多次采样投票、或确定性兜底策略。

### C. 数据 & 评测

**Q9: 你的评测集从哪来？100 张凭证够吗？**
> 金融欺诈的类型有几十种，100 张凭证可能连每种类型都覆盖不到。你计划怎么获取多样性足够的测试数据？有没有考虑用合成数据增强？
>
> **你的盲点**：方案提到了评测集的结构，但没有说明如何获取足够多样的真实欺诈案例。大部分 SME 不会把自己的欺诈案例分享给你。

**Q10: 如果客户问"误报率是多少"，你怎么回答？**
> V1 预估 30-40% 误报率——这意味着每标红 10 笔有 3-4 笔是冤枉的。CFO 看到第 5 个假警报可能就不再信任系统了。你的容忍阈值是多少？低于多少用户会流失？

**Q11: 你怎么防止评测集过拟合？**
> 如果你的 prompt 和规则都是针对 golden_set 调的，在真实数据上表现可能完全不同。你有留 holdout set 吗？有 cross-validation 策略吗？

### D. 商业 & 竞争

**Q12: MindBridge 融了 $4000 万做同样的事，你的壁垒是什么？**
> "面向 SME"和"飞书集成"不是壁垒——MindBridge 可以一周做个飞书插件。你的真正壁垒是什么？数据？算法？客户关系？监管认证？

**Q13: 为什么不直接做飞书的插件/应用，而要独立建站？**
> 飞书应用市场有天然流量。如果你的核心交互都在飞书里（通知、审批、表格），Next.js 前端的存在意义是什么？是不是可以纯做飞书应用 + 一个轻量设置页面？
>
> **你的盲点**：你同时建了 Next.js 前端 + 飞书集成，但没有明确主入口在哪里。如果用户两个地方都要看，体验反而更碎片化。

**Q14: 你的获客策略是什么？SME 的 CFO 在哪里？**
> 整个方案聚焦在产品功能，但没有一句话提到 GTM。SME 的 CFO 不刷 ProductHunt，不逛 GitHub。你怎么找到他们？会计师事务所渠道？代账公司合作？行业协会？

**Q15: 如果客户只用你的产品一次（年审时），你怎么保证留存？**
> 审计是低频行为。大部分 SME 一年审一次。如果客户只在审计季用你，ARR 很难做起来。你需要找到日常使用场景（日常报销审核？供应商尽调？）来提高使用频次。

## 四、你目前的 7 个盲点（按严重程度排序）

### 1. 没有定义「成功」的北极星指标
你列了 Precision/Recall 等技术指标，但没有产品指标。建议：
- **核心指标**：每月活跃扫描的凭证数（证明产品有持续价值）
- **价值指标**：AI 发现的异常中被用户确认为「真实问题」的数量
- **效率指标**：用户完成一轮审计所需时间（对比使用前）

### 2. 用户旅程中的「最后一公里」缺失
系统标出了风险，然后呢？CFO 看到红色标记后的下一步是什么？调查？联系供应商？修改账目？你的产品在「发现问题」之后就断了，但用户需要的是「发现 → 调查 → 处理 → 记录」的闭环。

### 3. 没有 Ground Truth 的冷启动问题
你默认客户有"已知的欺诈案例"可以用来验证 AI 的准确性。但大部分 SME 要么没发现过欺诈（所以不知道 AI 对不对），要么发现了但不愿意分享。V1 上线时，你怎么让客户相信"这个红色标记是真的"？

### 4. 多币种 / 跨境场景完全没考虑
你提到目标客户包括"经历跨境合规"的企业——但方案里没有一个字提到多币种处理、汇率换算、跨境发票格式差异（增值税 vs GST vs Sales Tax）。

### 5. 数据安全的方案不够具体
"本地部署"说了很多次，但 MVP 方案用的是 Supabase + Cloudflare R2 + Railway——全是云服务。如果一个金融客户问"我的发票数据存在哪里、谁能看到"，你怎么回答？

### 6. 没有竞品的 AI 能力对比
你列了 MindBridge/AppZen/Oversight 的定位差异，但没有对比它们的 AI 能力（Precision/Recall/支持的凭证类型/处理速度）。面试官会问：你凭什么说你的 AI 不比它们差？

### 7. 缺少 Prompt 作为核心资产的管理策略
你的审计能力核心是 prompt engineering——但方案里没有 prompt 版本管理、A/B 测试、回归测试的方案。Prompt 是你的"算法"，需要像代码一样管理。

## 五、建议的回答框架（面试用）

面对上述任何问题，使用这个结构：

```
1. 承认问题（展示自我认知）
   "这确实是一个关键问题，在 MVP 阶段我做了有意识的取舍..."

2. 解释取舍逻辑（展示产品判断力）
   "我选择先验证 X 假设，因为如果 X 不成立，Y 和 Z 都没意义..."

3. 给出演进路径（展示系统性思维）
   "在 V1 我的应对是 A，数据验证后 V2 升级为 B..."

4. 用指标说话（展示数据驱动）
   "我会通过跟踪 [具体指标] 来决定是否以及何时做这个改变..."
```
