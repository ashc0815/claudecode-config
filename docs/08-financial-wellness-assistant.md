# 财务健康助手 — 产品设计与技术架构

> **产品本质**：不是记账工具，是改善财务状态的陪伴式 AI 助手
> **核心差异**：记录 → 行动的闭环，解决"记了账但什么都没改变"的问题
> **技术路线**：单编排器 + 工具调用 + 分层记忆（MVP 不做多 Agent）
> **数据策略**：Local-first + Selective Cloud 加密

---

## 一、问题诊断：为什么现有记账产品都失败了

```
用户旅程中的四个断裂点：

          懒得记          不敢看          看不懂          改不动
            │                │               │               │
   ┌────────▼────────┐ ┌────▼────────┐ ┌────▼────────┐ ┌────▼────────┐
   │ 功能臃肿/广告多 │ │ 记账 = 焦虑  │ │ 消费/投资   │ │ 只有统计    │
   │ 每笔都要手动记  │ │ 发现超支更焦虑│ │ 数据割裂    │ │ 没有行动方案│
   │ 坚持不了3天     │ │ 变得抠搜/内疚 │ │ 看不到全貌  │ │ 知道问题    │
   │                 │ │              │ │             │ │ 但不知道怎么│
   │                 │ │              │ │             │ │ 改          │
   └─────────────────┘ └──────────────┘ └─────────────┘ └─────────────┘

现有产品为什么解决不了：
  - 随手记/挖财：工具思维，功能堆砌，解决"记"但不解决"改"
  - 支付宝账单：只有消费，没有投资/资产视图
  - 理财 App：只有投资，没有消费/预算联动
  - AI 记账 App：套壳 ChatGPT，没有真正理解财务语境

★ 核心洞察：用户不需要更好的记账工具，
   需要的是"有人帮我看着钱、告诉我该怎么做、陪我做到"
```

### 关键设计原则

```
1. 每次只给一个行动（One Action Rule）
   ✗ "你应该减少外卖、控制打车、取消不用的订阅、每月存2000..."
   ✓ "这周试试周二带饭？就这一件事。"
   → 减少决策负担，提高完成率

2. 建议必须具体可执行（Actionable Specificity）
   ✗ "减少不必要支出"
   ✓ "取消你3个月没用的 Spotify 会员（每月省 ¥15）"
   → 用户不需要思考"怎么做"，直接做

3. 不评判，只陪伴（Non-judgmental Companionship）
   ✗ "你又超支了" / "你的消费习惯很差"
   ✓ "这个月花得比平时多一些，看看是什么原因？"
   → 降低焦虑，避免用户回避

4. 渐进式信息披露（Progressive Disclosure）
   - 首次使用：只看本月消费概览
   - 第2周：引入趋势对比
   - 第1个月：开始看资产全貌
   - 第3个月：引入投资组合分析
   → 避免信息过载导致焦虑

5. 失败是信号，不是问题（Failure as Signal）
   - 用户没完成承诺 → 不是"用户执行力差"
   - 而是"建议不够适配" → 换个更容易的
   → 系统持续学习什么对这个用户有效
```

## 四、功能模块详解

### 模块 A：懒人记录（解决"懒得记"）

```
┌────────────────────────────────────────────────────────┐
│               Lazy Recording System                     │
│                                                         │
│  方式 1：银行/支付账单导入（主力）                         │
│  ├── 支持导入支付宝/微信账单 CSV/PDF                      │
│  ├── 支持导入银行账单（招行/工行等主流银行）                │
│  ├── AI 自动分类（吃饭/交通/娱乐/...）                    │
│  ├── 合并重复（微信→美团外卖 = 一笔餐饮）                  │
│  └── 频率：每周或每月导入一次即可                          │
│                                                         │
│  方式 2：基金/理财净值自动更新                             │
│  ├── 输入持仓基金代码 → 自动拉取每日净值                   │
│  ├── 支持：公募基金、ETF（通过公开 API）                   │
│  ├── 自动计算收益率、浮盈浮亏                              │
│  └── 与消费数据联动 → 资产全貌                             │
│                                                         │
│  方式 3：快速手记（补充）                                  │
│  ├── 语音："今天午饭花了32块"                               │
│  ├── 拍照：拍小票自动识别                                  │
│  └── 快捷记录：常用场景一键记录                             │
│                                                         │
│  ★ 核心思路：                                             │
│  不要求用户实时记录每一笔                                  │
│  而是定期批量导入 + AI 补全                                │
│  "记录"不是目的，"有数据可分析"才是目的                     │
└────────────────────────────────────────────────────────┘
```

│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  我们的助手说：                                    │   │
│  │                                                   │   │

│  焦虑缓冲的具体策略：                                     │
│                                                         │

│                                                         │

│     "比上月少花了 ¥200，继续保持"                         │

## 八、关键决策日志

| 决策 | 选择 | 放弃方案 | 为什么 |
|---|---|---|---|
| MVP 架构 | 单编排器 + 工具 | 多 Agent | MVP 阶段复杂度不需要多 Agent，单编排器 + Function Calling 够用 |
| 记录方式 | 账单导入为主 | 实时手动记录 | 解决"懒得记"，降低使用门槛 |
| AI 表达风格 | 焦虑缓冲 | 直接报数据 | 核心差异化，解决"不敢看" |
| 建议策略 | 每次一个行动 | 多建议列表 | 减少决策瘫痪，提高行动率 |
| 存储 | Local SQLite | Cloud-first | 数据敏感度高，用户信任优先 |
| 投资功能 | MVP 不做 | 一开始做 | 消费侧闭环是最小可验证单元 |
