# AI Anti-Fraud 产品规划 — 聚焦三大 AI 差异化能力

> **产品定位**：中国市场的 AppZen / Oversight 竞品 — 异步后置审计层
> **核心差异**：规则引擎做不好的、只有 AI 才能解决的三个场景
> **架构模式**：异步后置检测（用户正常提交 → 后台 AI 批量审计 → 问题推送审计员）
> **目标客户**：在华跨国企业 + 中大型民企（使用 Concur/SAP/Oracle 等 ERP）

---

## 核心战略转变

```
旧定位（已否定）：
  Concur 前置智能审核层
  ├── 前端验证延迟太高（每张几分钟，UX 极差）
  ├── 中国场景规则（打车/餐费/发票抬头）→ TMC 和财务系统已解决
  └── 规则校验 ≠ AI 差异化（if-else 就够了）

新定位：
  异步后置 AI 审计平台（中国版 AppZen/Oversight）
  ├── 用户正常提交报销（不阻塞流程，0 延迟感知）
  ├── 后台 AI 批量跑检测（分钟级 → 用户无感）
  ├── 聚焦规则引擎做不了的 AI 场景
  └── 结果推送给审计员/财务 Dashboard
```

### 为什么是异步后置？

```
AppZen 的模式 = "pre-reimbursement, post-submission"
  用户提交 → 正常进入审批流 → AI 在后台检测 → 付款前拦截问题件

这解决了你同事提到的延迟问题：
  前端同步验证：每张 3-5 分钟 → 用户体验灾难
  后台异步审计：批量处理，用户无感 → 审计员只看问题件
```

> 这三个场景的共同特征：**规则引擎做不好，传统抽样发现不了，只有 AI 能规模化解决**

### 能力一：虚假发票 / 篡改发票检测（图像取证 AI）

#### 问题定义

```
中国发票欺诈的现状：
  ├── 税局验真只能验"这张发票是否存在"
  │   → 无法检测：真发票被 PS 改金额、真发票张冠李戴
  │
  ├── OCR 只做文字提取
  │   → 无法检测：图像级篡改（改数字、拼接、AI 生成）
  │
  └── 人工抽查覆盖率 < 10%
      → 小额篡改完全漏网

真实欺诈手法（按检测难度排序）：
  Level 1: 重复报销同一张发票 ← 规则引擎能做（hash 查重）
  Level 2: 用别人的发票报销 ← 规则能做（抬头/税号校验）
  Level 3: PS 修改发票金额 ← 需要图像取证 AI ★
  Level 4: 拼接多张发票内容 ← 需要图像取证 AI ★
  Level 5: AI 生成完全虚假的发票 ← 需要图像取证 AI ★
  Level 6: 真发票+假消费（空刷/代开） ← 需要行为分析（能力二）
```

#### AI 技术方案

```
┌─────────────────────────────────────────────────────┐
│          Invoice Forensics AI Pipeline               │
│                                                      │
│  Input: 发票图片/PDF                                  │
│                                                      │
│  Stage 1: 基础验证（确定性，非 AI）                     │
│  ├── 发票号/代码 → 税局 API 验真                       │
│  ├── Hash 指纹 → 查重库比对                            │
│  └── 发票抬头/税号 → 与企业信息匹配                     │
│  → 这一步过滤 ~60% 的已知问题                          │
│                                                      │
│  Stage 2: 图像取证分析（AI 核心）                       │
│  ┌──────────────────────────────────────────────┐    │
│  │                                              │    │
│  │  2a. 像素级篡改检测                           │    │
│  │  ├── ELA (Error Level Analysis)              │    │
│  │  │   → 检测 JPEG 压缩不一致区域               │    │
│  │  ├── 噪声分析 (Noise Analysis)               │    │
│  │  │   → 不同来源的图像噪声模式不一致            │    │
│  │  ├── 边缘检测 (Edge Detection)               │    │
│  │  │   → PS 修改后的边缘 vs 原始打印边缘不同     │    │
│  │  └── 字体一致性检测                           │    │
│  │      → 同一张发票上出现不同字体/字号           │    │
│  │                                              │    │
│  │  2b. 多模态 LLM 验证                          │    │
│  │  ├── Claude Vision: 发票整体合理性判断         │    │
│  │  │   → "这张发票的排版/字体/印章位置是否正常？" │    │
│  │  ├── 金额-明细一致性                          │    │
│  │  │   → OCR 提取的明细项加总 vs 总金额是否一致   │    │
│  │  ├── 印章真实性初筛                           │    │
│  │  │   → 印章位置、清晰度、与公司名匹配度        │    │
│  │  └── AI 生成发票检测                          │    │
│  │      → 检测 Midjourney/DALL-E 生成的假发票     │    │
│  │      → 特征：过于完美的对齐、缺少打印噪声      │    │
│  │                                              │    │
│  │  2c. 跨发票交叉验证                           │    │
│  │  ├── 同一供应商的发票模板是否一致               │    │
│  │  ├── 发票号码序列是否连续合理                  │    │
│  │  └── 同期发票的印刷批次特征是否一致             │    │
│  │                                              │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
│  Stage 3: 风险评分 + 证据输出                          │
│  ├── 综合评分: 0-100 (各检测维度加权)                  │
│  ├── 可视化证据: 标注可疑区域 (热力图)                  │
│  ├── 自然语言解读: "金额区域检测到图像编辑痕迹..."       │
│  └── 置信度 + 建议动作                                │
│                                                      │
│  Output → 审计员 Dashboard                            │
│  ├── 高风险: 自动冻结报销 + 推送审计员                  │
│  ├── 中风险: 标记需关注 + 排入审核队列                  │
│  └── 低风险: 正常放行                                  │
└─────────────────────────────────────────────────────┘
```

#### 技术选型

```
图像取证层:
  ├── Pillow + OpenCV: ELA/噪声分析/边缘检测（开源，成熟）
  ├── 预训练模型: 图像篡改检测 (ManTraNet / MVSS-Net)
  │   → 学术界成熟方案，可直接部署
  └── Claude Vision: 整体合理性 + AI 生成检测

成本估算（每张发票）:
  ├── Stage 1 (验真/查重): ~¥0（自有服务）
  ├── Stage 2a (图像分析): ~¥0.01（本地推理）
  ├── Stage 2b (Claude Vision): ~¥0.10-0.15
  ├── Stage 2c (交叉验证): ~¥0.02（数据库查询）
  └── 总计: ~¥0.15/张
```

#### 与竞品对比

```
Oversight "Receipt Analytics":
  ├── 宣称 95%+ 准确率检测篡改/AI 生成收据
  ├── 主要面向英文 Receipt
  └── 不支持中国增值税发票体系

你的差异化:
  ├── 针对中国增值税发票版式优化（版式固定 → 检测更精准）
  ├── 结合税局验真 API（Oversight 没有中国税务接口）
  ├── 支持电子发票 XML 校验（区块链发票 hash 验证）
  └── 中国场景特有: 代开发票检测、空刷 POS 发票检测
```

### 能力二：报销行为异常模式识别（跨时间维度）

#### 问题定义

```
单笔审核 vs 行为模式分析:

  规则引擎能做的（单笔维度）:
    "这笔打车费 ¥298 < 上限 ¥300" → PASS
    "发票日期在报销期内" → PASS
    "发票验真通过" → PASS

  规则引擎做不了的（跨时间维度）:
    "这个员工连续 3 个月每周 5 笔打车费都在 ¥295-299" → 阈值试探
    "这个员工上月报销突增 300%，但工作量没变" → 行为突变
    "这个员工和某供应商的交易频率远超同部门同事" → 异常关联
    "这个员工每到月底就突击提交大量报销" → 时间模式异常

  核心 insight:
    每一笔单独看都合规，但放在一起看就不正常
    这正是 AppZen/Oversight 的核心卖点之一
```

│           Behavioral Anomaly Detection Engine               │

│  数据输入层                                                   │

│  ├── HR 数据（部门/职级/入职时间/直属上级）                      │
│  └── 历史审计结果（已确认异常/误报标记）                         │
│                                                             │
│  ═══════════════════════════════════════════════             │
│                                                             │

│  │                                                   │      │
│  │  个人基线（纵向对比）                               │      │

│  │  └── 例: 同部门 10 人月均 ¥3K，此人 ¥12K            │      │
│  │                                                   │      │

│                                                             │
│  分析层 3: AI 解读 + 报告生成                                 │
│  ┌──────────────────────────────────────────────────┐      │
│  │  Claude: 将统计异常转化为审计员可理解的叙述             │      │
│  │                                                   │      │
│  │  输入: 统计信号 + 原始数据 + 员工上下文               │      │
│  │  输出:                                             │      │
│  │  ├── 异常摘要: "张三近 90 天的交通费报销呈现          │      │
│  │  │    阈值试探模式，12 笔中 10 笔在 ¥295-299..."     │      │
│  │  ├── 建议调查方向: "建议核查实际出行记录"             │      │
│  │  ├── 风险评级: HIGH / MEDIUM / LOW                  │      │
│  │  └── 类似历史案例: "2025Q3 曾有类似模式被确认为欺诈"   │      │
│  └──────────────────────────────────────────────────┘      │
│                                                             │

### 能力三：供应商关联关系图谱（员工-供应商利益输送）

#### 问题定义

```
最难检测的 fraud 类型 = 内外勾结:

  员工和供应商串通:
    ├── 员工注册关联公司 → 自己审批自己的采购
    ├── 员工配偶/亲属开公司 → 成为公司供应商
    ├── 多个员工集体向同一"冷门"供应商报销
    └── 供应商给员工回扣 → 员工集中采购

  为什么规则引擎抓不到:
    ├── 每笔交易单独看完全合规
    ├── 关联关系隐藏在工商信息/地址/银行账号中
    └── 需要"连线"才能看出模式

  ACFE 2024 报告:
    资产侵占 (Asset Misappropriation) 占所有 fraud 的 86%
    其中 billing schemes（虚假供应商）是第二大类型
    中位损失: $100,000 / 案件
```

#### 技术选型

```
图数据库:
  ├── Neo4j (成熟, 社区版免费)
  │   → 适合复杂关系查询 (Cypher 查询语言)
  │   → 内置社区发现算法 (GDS 库)
  └── 备选: NetworkX (轻量, 纯 Python, MVP 阶段足够)

外部数据接入:
  ├── 天眼查 API: 工商信息/股东/法人/变更历史
  │   → 定价: 按查询次数，~¥0.5-2/次
  ├── 企查查 API: 类似天眼查，价格竞争
  └── 国家企业信用信息公示系统: 免费但需要爬取

图分析算法:
  ├── 最短路径: 发现员工-供应商间的最短关联
  ├── PageRank: 识别图中"关键节点"（高风险枢纽）
  ├── Louvain 社区发现: 找到异常紧密的子图
  └── 中心性分析: 找到异常多连接的供应商/员工

AI 层:
  └── Claude: 将图分析结果 → 可读的调查报告
```
