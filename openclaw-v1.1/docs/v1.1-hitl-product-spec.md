# OpenClaw V1.1 — Human-in-the-Loop (HITL) 产品规格书

> V1.1 核心命题：让 AI 和人协同进化——AI 做 90% 的苦活，人做 10% 的判断，人的每个判断反过来让 AI 更准。

---

## 〇、V1 → V1.1 的关键跃迁

```
V1（已有）                              V1.1（本文档）
─────────────────────────              ─────────────────────────
✅ 单管线：OCR→Claude→规则→飞书        ✅ 同架构，但加入闭环反馈
✅ 飞书三按钮（确认/误报/调查）          ✅ 反馈数据驱动规则自调优
✅ 10 条硬编码规则                      ✅ 规则置信度动态调整
✅ 100 张评测集                         ✅ 500+ 生产数据自动标注
✅ Supabase 3 张表                     ✅ 加 3 张 HITL 表（共 6 张）
❌ 用户反馈只记录不使用                  ✅ 反馈→规则参数→prompt 自动优化
❌ 误报率 30-40%                        ✅ 目标降至 ≤15%
❌ 无法衡量系统是否在变好                 ✅ 周度 Precision/Recall 追踪
```

### V1.1 不做的事（依然极简）
```
❌ 不加前端 Dashboard（继续用飞书 + 飞书多维表格）
❌ 不引入 Redis Queue（BackgroundTasks 继续够用）
❌ 不做多 Agent 编排（V2 再做）
❌ 不做海外凭证 OCR（V2 再做）
❌ 不做多币种（继续只支持 CNY）
❌ 不自建 ML 模型（继续用 Claude API）
```

### 开发量预估
```
V1.1 增量代码: ~600 行 Python（在 V1 的 1500 行基础上）
新增依赖: 无（继续用 fastapi, httpx, supabase-py）
开发周期: 2-3 周
新增成本: $0（只是更聪明地用现有数据）
```

```sql
-- ============================================
-- 表 1: audit_results（V1 已有，V1.1 扩展）
-- ============================================
CREATE TABLE audit_results (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  audit_id      TEXT UNIQUE NOT NULL,       -- "AUD-2026-0312-0847"
  created_at    TIMESTAMPTZ DEFAULT NOW(),

  -- 凭证信息
  document_url  TEXT NOT NULL,              -- Supabase Storage URL
  ocr_raw_text  TEXT,                       -- OCR 原始文本
  ocr_structured JSONB NOT NULL,            -- 结构化 OCR 输出
  ocr_confidence FLOAT,                     -- OCR 置信度

  -- 审计结果
  risk_level    TEXT NOT NULL,              -- "pass" | "warn" | "fail"
  risk_score    INTEGER,                    -- 0-100 综合风险评分
  risk_flags    JSONB NOT NULL,             -- 风险标记数组
  ai_reasoning  TEXT,                       -- Claude 推理过程

  -- V1.1 新增字段
  prompt_version TEXT DEFAULT 'v1.0',       -- 使用的 prompt 版本
  rule_params_snapshot JSONB,               -- 本次使用的规则参数快照
  processing_time_ms INTEGER,               -- 处理耗时

  -- 状态
  status        TEXT DEFAULT 'pending_review', -- "pending_review" | "confirmed" | "false_positive" | "investigating"
  resolved_at   TIMESTAMPTZ
);

-- 索引
CREATE INDEX idx_audit_results_status ON audit_results(status);
CREATE INDEX idx_audit_results_risk ON audit_results(risk_level);
CREATE INDEX idx_audit_results_created ON audit_results(created_at);

-- ============================================
-- 表 2: feedback（V1 已有，V1.1 大幅扩展）
-- ============================================
CREATE TABLE feedback (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  audit_id        TEXT NOT NULL REFERENCES audit_results(audit_id),
  created_at      TIMESTAMPTZ DEFAULT NOW(),

  -- 反馈核心
  action          TEXT NOT NULL,              -- "confirmed" | "false_positive" | "investigate"
  reviewer_id     TEXT NOT NULL,
  reviewer_role   TEXT,

  -- V1.1 新增: 结构化反馈
  false_positive_reason TEXT,                -- 误报原因分类码
  corrected_risk_level  TEXT,                -- 用户认为的真实风险
  per_flag_feedback     JSONB,               -- 逐条规则反馈
  free_text_note        TEXT,                -- 自由文本备注
  time_spent_seconds    INTEGER,             -- 决策耗时

  -- V1.1 新增: 学习标记
  used_for_learning BOOLEAN DEFAULT FALSE,   -- 是否已被 learner 消费
  learning_batch_id TEXT                     -- 属于哪个学习批次
);

CREATE INDEX idx_feedback_audit ON feedback(audit_id);
CREATE INDEX idx_feedback_action ON feedback(action);
CREATE INDEX idx_feedback_learning ON feedback(used_for_learning);

-- ============================================
-- 表 3: audit_log（V1 已有，不变）
-- ============================================
CREATE TABLE audit_log (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  event_type  TEXT NOT NULL,     -- "upload" | "ocr" | "audit" | "feedback" | "rule_update"
  actor       TEXT NOT NULL,     -- "system" | user_id
  audit_id    TEXT,
  details     JSONB NOT NULL,    -- 事件详情
  -- 不可变: 无 UPDATE/DELETE 权限
  CONSTRAINT no_update CHECK (TRUE)
);

CREATE INDEX idx_audit_log_event ON audit_log(event_type);
CREATE INDEX idx_audit_log_audit ON audit_log(audit_id);

-- ============================================
-- 表 4: rule_params（V1.1 新增）
-- 规则参数的动态配置，替代硬编码
-- ============================================
CREATE TABLE rule_params (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rule_name       TEXT NOT NULL,              -- "threshold_proximity"
  param_name      TEXT NOT NULL,              -- "threshold_amount"
  param_value     JSONB NOT NULL,             -- 50000

  -- 来源追踪
  source          TEXT NOT NULL,              -- "manual" | "auto_tuned" | "default"
  previous_value  JSONB,                      -- 调整前的值
  adjustment_reason TEXT,                     -- "基于 47 条反馈，周末发票误报率 62%"

  -- 版本
  version         INTEGER DEFAULT 1,
  effective_from  TIMESTAMPTZ DEFAULT NOW(),
  effective_until TIMESTAMPTZ,                -- NULL = 当前生效
  created_by      TEXT DEFAULT 'system',

  UNIQUE(rule_name, param_name, version)
);

CREATE INDEX idx_rule_params_active ON rule_params(rule_name, effective_until)
  WHERE effective_until IS NULL;

-- ============================================
-- 表 5: prompt_versions（V1.1 新增）
-- Prompt 版本管理 + A/B 测试
-- ============================================
CREATE TABLE prompt_versions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  created_at      TIMESTAMPTZ DEFAULT NOW(),

  -- 性能追踪
  total_uses      INTEGER DEFAULT 0,
  confirmed_count INTEGER DEFAULT 0,          -- 用户确认的次数
  false_positive_count INTEGER DEFAULT 0,     -- 误报次数
  precision       FLOAT,                      -- confirmed / (confirmed + false_positive)

  -- 状态
  status          TEXT DEFAULT 'active',      -- "active" | "testing" | "retired"
  traffic_pct     INTEGER DEFAULT 100,        -- A/B 测试流量百分比
  notes           TEXT
);

-- ============================================
-- 表 6: weekly_metrics（V1.1 新增）
-- 周度指标快照，追踪系统是否在变好
-- ============================================
CREATE TABLE weekly_metrics (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  week_start      DATE NOT NULL,              -- 周一日期
  created_at      TIMESTAMPTZ DEFAULT NOW(),

  -- 量级指标
  total_audits    INTEGER,                    -- 本周处理总量
  pass_count      INTEGER,
  warn_count      INTEGER,
  fail_count      INTEGER,

  -- 质量指标（核心！）
  feedback_count  INTEGER,                    -- 本周收到的反馈数
  confirmed_anomalies INTEGER,                -- 确认异常数
  false_positives INTEGER,                    -- 误报数
  precision       FLOAT,                      -- 确认 / (确认 + 误报)

  -- 反馈覆盖率
  feedback_rate   FLOAT,                      -- 有反馈的 / 总标红数

  -- 效率指标
  avg_review_time_seconds FLOAT,              -- 平均人工审核耗时
  avg_processing_time_ms  FLOAT,              -- 平均系统处理耗时

  -- 规则命中分布
  rule_trigger_distribution JSONB,            -- {"threshold_proximity": 23, "weekend": 15, ...}
  false_positive_by_rule    JSONB,            -- {"threshold_proximity": 8, "weekend": 12, ...}

  -- Prompt 版本
  prompt_version_used TEXT,

  UNIQUE(week_start)
);
```

## 四、HITL 学习循环（learner.py 核心逻辑）

### 4.1 学习循环总览

```
每周日晚 23:00 自动运行（cron job / Supabase Edge Function）

┌──────────────────────────────────────────────────────────┐
│                   Weekly Learning Cycle                   │
│                                                          │
│   Step 1: 收集本周反馈                                    │
│   ┌──────────────────────────────────────────────────┐   │
│   │ SELECT * FROM feedback                           │   │
│   │ WHERE used_for_learning = FALSE                  │   │
│   │ AND action IN ('confirmed', 'false_positive')    │   │
│   └───────────────────────┬──────────────────────────┘   │
│                           │                              │
│   Step 2: 按规则聚合误报率                                │
│   ┌───────────────────────▼──────────────────────────┐   │
│   │ 每条规则的本周表现:                                │   │
│   │                                                   │   │
│   │ threshold_proximity:  触发 23 次, 误报 8 次 = 65%  │   │
│   │ weekend_invoice:      触发 15 次, 误报 12 次 = 20% │   │
│   │ vendor_frequency:     触发 31 次, 误报 5 次 = 84%  │   │
│   │ round_amount:         触发 12 次, 误报 9 次 = 25%  │   │
│   └───────────────────────┬──────────────────────────┘   │
│                           │                              │
│   Step 3: 自动调优决策                                    │
│   ┌───────────────────────▼──────────────────────────┐   │
│   │                                                   │   │
│   │ IF 规则精准率 < 40%（连续 2 周）:                    │   │
│   │   → 降低该规则权重（risk_score 贡献 × 0.5）         │   │
│   │   → 生成告警: "weekend_invoice 规则建议关闭/调整"    │   │
│   │   → 飞书通知管理员                                  │   │
│   │                                                   │   │
│   │ IF 规则精准率 50%-70%:                              │   │
│   │   → 微调参数（如阈值 ±10%）                          │   │
│   │   → 记录调整日志到 rule_params                      │   │
│   │                                                   │   │
│   │ IF 规则精准率 > 80%:                                │   │
│   │   → 保持不变，标记为高可靠规则                        │   │
│   │                                                   │   │
│   │ ⚠️ 安全边界:                                       │   │
│   │   → 规则权重最低降至 0.2（不自动删除规则）            │   │
│   │   → 每周最多调整 3 条规则                            │   │
│   │   → 所有调整需管理员飞书确认后生效                    │   │
│   └───────────────────────┬──────────────────────────┘   │
│                           │                              │
│   Step 4: Prompt 迭代建议                                 │
│   ┌───────────────────────▼──────────────────────────┐   │
│   │                                                   │   │
│   │ 收集本周所有 false_positive 的 free_text_note      │   │
│   │ → 输入 Claude:                                    │   │
│   │   "以下是本周 12 条误报的用户备注，                   │   │
│   │    请分析模式，建议 prompt 如何调整"                  │   │
│   │                                                   │   │
│   │ → Claude 输出 prompt 调整建议                      │   │
│   │ → 生成新 prompt 版本 (v1.1-candidate)              │   │
│   │ → 下周以 20% 流量 A/B 测试                         │   │
│   │                                                   │   │
│   └───────────────────────┬──────────────────────────┘   │
│                           │                              │
│   Step 5: 写入周度指标快照                                │
│   ┌───────────────────────▼──────────────────────────┐   │
│   │ INSERT INTO weekly_metrics (...)                  │   │
│   │ → 飞书发送「周报」卡片给管理员                       │   │
│   └──────────────────────────────────────────────────┘   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 4.2 规则参数动态调优逻辑

```python
# learner.py 核心伪代码

def weekly_learning_cycle():
    """每周执行一次的学习循环"""

    # Step 1: 拉取未消费的反馈
    feedbacks = db.query("""
        SELECT f.*, ar.risk_flags, ar.prompt_version
        FROM feedback f
        JOIN audit_results ar ON f.audit_id = ar.audit_id
        WHERE f.used_for_learning = FALSE
        AND f.action IN ('confirmed', 'false_positive')
    """)

    if len(feedbacks) < 10:
        # 样本不够，跳过本周
        notify_admin("本周反馈不足 10 条，跳过自动调优")
        return

    # Step 2: 按规则聚合
    rule_stats = {}
    for fb in feedbacks:
        for flag in fb.per_flag_feedback or []:
            rule = flag["rule"]
            if rule not in rule_stats:
                rule_stats[rule] = {"agree": 0, "disagree": 0, "reasons": []}
            if flag["agree"]:
                rule_stats[rule]["agree"] += 1
            else:
                rule_stats[rule]["disagree"] += 1
                if flag.get("note"):
                    rule_stats[rule]["reasons"].append(flag["note"])

    # Step 3: 生成调优建议
    adjustments = []
    for rule, stats in rule_stats.items():
        total = stats["agree"] + stats["disagree"]
        if total < 5:
            continue  # 样本不够，跳过

        precision = stats["agree"] / total

        if precision < 0.4:
            adjustments.append({
                "rule": rule,
                "action": "reduce_weight",
                "new_weight": 0.5,
                "reason": f"精准率 {precision:.0%}（{stats['agree']}/{total}），建议降权",
                "user_reasons": stats["reasons"][:3]
            })
        elif precision < 0.7:
            adjustments.append({
                "rule": rule,
                "action": "fine_tune",
                "reason": f"精准率 {precision:.0%}，建议微调参数"
            })

    # Step 4: 发送飞书确认（HITL — 人确认后才生效！）
    if adjustments:
        send_adjustment_proposal(adjustments)
        # 管理员在飞书点击 [批准] 后才执行 apply_adjustments()

    # Step 5: 写入周度快照
    save_weekly_metrics(feedbacks, rule_stats)

    # Step 6: 标记反馈已消费
    mark_feedbacks_consumed(feedbacks)


def apply_adjustments(approved_adjustments):
    """管理员批准后执行调整"""
    for adj in approved_adjustments:
        # 保存旧参数
        old_params = get_current_params(adj["rule"])

        # 写入新参数（带版本号）
        db.insert("rule_params", {
            "rule_name": adj["rule"],
            "param_name": "weight",
            "param_value": adj["new_weight"],
            "source": "auto_tuned",
            "previous_value": old_params,
            "adjustment_reason": adj["reason"],
            "version": old_params["version"] + 1
        })

        # 不可变审计日志
        db.insert("audit_log", {
            "event_type": "rule_update",
            "actor": "learner_v1.1",
            "details": {
                "rule": adj["rule"],
                "old_weight": old_params["weight"],
                "new_weight": adj["new_weight"],
                "reason": adj["reason"],
                "approved_by": "admin_xxx"
            }
        })
```

def evaluate_rules(expense: dict) -> dict:
    """执行所有规则，返回结构化结果"""

    triggered = []
    total_score = 0

    for rule_name, rule in RULES.items():
        # 从数据库加载最新参数（覆盖默认值）
        params = load_rule_params(rule_name, defaults=rule["params"])

        if rule["check"](expense, params):
            weight = params.get("weight", 1.0)
            severity_score = {"critical": 40, "high": 25, "medium": 15, "low": 8}
            score = severity_score[rule["severity"]] * weight

            triggered.append({
                "rule": rule_name,
                "description": rule["description"],
                "severity": rule["severity"],
                "weight": weight,
                "score": round(score),
                "confidence": calculate_rule_confidence(rule_name),
            })
            total_score += score

    # 分流决策
    risk_level = "pass"
    if any(t["severity"] == "critical" for t in triggered):
        risk_level = "fail"
    elif total_score >= 50:
        risk_level = "fail"
    elif total_score >= 20:
        risk_level = "warn"

    return {
        "risk_level": risk_level,
        "risk_score": min(total_score, 100),
        "triggered_rules": triggered,
    }


def calculate_rule_confidence(rule_name: str) -> float:
    """基于历史反馈计算规则置信度"""
    stats = db.query("""
        SELECT
            COUNT(*) FILTER (WHERE f.action = 'confirmed') as confirmed,
            COUNT(*) FILTER (WHERE f.action = 'false_positive') as fp
        FROM feedback f
        JOIN audit_results ar ON f.audit_id = ar.audit_id
        WHERE ar.risk_flags @> %s
        AND f.created_at > NOW() - INTERVAL '30 days'
    """, [json.dumps([{"rule": rule_name}])])

    total = stats["confirmed"] + stats["fp"]
    if total < 5:
        return 0.7  # 数据不足时默认 70%
    return round(stats["confirmed"] / total, 2)

## 十、V1.1 关键设计决策 Q&A

```
Q: 为什么不直接用 ML 模型而是规则调权重？
A: V1.1 阶段数据量不够训练 ML。规则调权重的好处是：
   1. 可解释（"周末规则降权是因为 62% 误报"）
   2. 可回滚（改回 weight=1.0 即可）
   3. 人可以理解和批准（HITL 的核心）
   V2 积累够数据后再上 ML。

Q: 为什么学习循环是周度而不是实时？
A: 三个原因：
   1. 单次反馈可能有偏（某个用户习惯性点"确认"）
   2. 规则频繁变动会让用户困惑（"上周还标红，这周不标了？"）
   3. 需要管理员批准，实时批准不现实

Q: 自动调优会不会把规则调坏？
A: 有三层安全机制：
   1. 每条规则有 min_weight（如 duplicate_invoice 最低 1.0，不可降权）
   2. 每周最多调整 3 条规则
   3. 所有调整需管理员飞书点击 [批准] 后才生效（HITL！）

Q: Prompt A/B 测试怎么保证一致性？
A: 同一张发票只用一个 prompt 版本处理。
   版本标记写入 audit_results.prompt_version 字段。
   周度对比两个版本的精准率，样本 > 50 后做统计显著性检验。

Q: 反馈覆盖率不够怎么办？
A: 三个策略：
   1. 飞书卡片 24 小时后自动提醒（"您有 3 笔审计结果待确认"）
   2. 精简反馈流程——点一个按钮就算反馈（不强制填原因）
   3. 把反馈率作为客户健康度指标，低于 50% 时客户成功主动跟进

Q: V1.1 的数据怎么为 V2 铺路？
A: V1.1 积累的三类数据直接喂 V2：
   1. per_flag_feedback → V2 规则引擎的决策树训练数据
   2. false_positive_reason 分类 → V2 上下文聚合器的特征工程
   3. prompt_versions 的 A/B 测试结果 → V2 多 Agent 的 prompt 基线
   4. weekly_metrics → V2 异常检测的行为基线
```

## 十一、面试话术（V1.1 版补充）

### 被问「你怎么保证 AI 审计的准确率？」

```
"我不保证——这正是产品设计的核心。

 V1 上线时精准率只有 60%，10 笔标红里 4 笔是误报。
 但 V1.1 加了一个闭环：

 每次财务人员点击 [确认] 或 [误报]，
 系统记录这个反馈，每周自动分析：
 · 哪条规则误报最多 → 降权
 · 哪类误报重复出现 → 调整 prompt
 · 精准率是否在上升 → 周报自动发给管理员

 3 周后精准率从 60% 涨到 72%。
 不是因为我调了代码，
 是因为用户的每一次点击都在教 AI 什么是'这家公司的正常'。

 这就是 Human-in-the-Loop：
 不是让 AI 完全自主，
 而是让人的判断力和 AI 的覆盖率结合。
 AI 看 100% 的凭证，人只审 25% 的标红——
 还顺便教会了 AI 下次少犯同样的错。"
```

### 被问「这个 HITL 跟 MindBridge 有什么区别？」

```
"MindBridge 是给审计师用的——他们的 HITL 是 '审计师在 dashboard 里
 review 异常然后写审计底稿'。专业、强大、$50K/年。

 我的 HITL 是给 SME 财务人员用的——
 他们的 HITL 就是在飞书里点一个按钮: [确认] 或 [误报]。

 不需要审计师资质。
 不需要学新系统。
 不需要写审计底稿。

 就像 Notion 之于 Confluence：
 同样是文档工具，但 Notion 让非技术人员也能用好。
 我做的是审计领域的 'Notion 体验'——
 把 HITL 的门槛从 '审计师在 Dashboard 里标注'
 降到 '财务在飞书里点按钮'。"
```
