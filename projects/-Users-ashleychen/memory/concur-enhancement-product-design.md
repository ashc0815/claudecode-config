# OpenClaw B2B Finance — Concur 智能审核增强层产品设计

> **产品本质**：SAP Concur 的前置智能审核层，面向在华跨国企业
> **核心定位**：员工报销的第一个 contact point → 审核通过后才进入 Concur
> **目标客户**：使用 Concur 的跨国企业（医药医疗器械、零售如味好美/PVH），几百到几千人
> **核心壁垒**：中国最懂 Concur + 最懂中国财税政策 + 已有实施服务和客户基础

### 关键数据流

```
上行（员工 → Concur）：
  员工上传凭证到你们系统
    → OCR 识别 + 发票验真查重（中国发票调税局接口）
    → 细粒度规则校验（非出差晚9点后打车、电话号码≤2个等）
    → 通过 → 结构化数据通过 Concur API 写入 Concur
    → 不通过 → 打回给员工，附带原因

下行（Concur → 你们系统）：
  Concur 结构化数据（金额/日期/费用类型/出差申请单/审批状态）
    → 通过 Concur API 拉取
    → 与你们系统中的凭证原件做交叉核对
    → 结果展示在财务 Dashboard

事后（财务 Dashboard）：
  财务同事在你们 Dashboard 上操作
    → 发现问题 → 通过 Concur API 打回报销单给员工
    → 异常检测报告 → 给内审/CFO
```

## 二、核心功能模块拆解

### 模块 A：细粒度规则引擎（Concur 做不了的）

**这是核心差异化——Concur 只能做"等级不同限额不同"，你们做的是语义级审核。**

```
规则分类体系：

1. 场景条件规则（Concur 完全做不了）
   ├── 非出差情况下，晚 9 点后打车/叫餐才可报销
   ├── 出差申请单与实际行程不匹配（日期/城市/天数）
   ├── 周末/节假日报销需关联加班审批单
   └── 跨城市交通费用需有对应出差申请

2. 累计/频次规则（Concur 做不好）
   ├── 一个员工仅可报销 2 个电话号码
   ├── 同一供应商月度累计超过阈值
   ├── 同一费用类型月度报销次数异常
   └── 单日多笔同类费用（拆单检测）

3. 凭证-报销项匹配规则
   ├── 发票金额与报销金额是否一致
   ├── 发票日期是否在报销期间范围内
   ├── 发票抬头是否为公司名称
   ├── 发票项目与费用类型是否匹配
   └── 水单/信用卡账单金额与报销金额核对

4. 合规性规则
   ├── 中国增值税发票验真（税局接口）
   ├── 发票查重（同一张发票不能报两次）
   ├── 发票是否过期（超过报销有效期）
   └── 海外发票基本信息完整性（金额/日期/商户名）
```

**规则引擎架构设计（重点展示AI+架构能力）：**

```
┌────────────────────────────────────────────────┐
│              Rule Engine Architecture           │
│                                                 │
│  ┌───────────────┐    ┌─────────────────────┐  │
│  │ Rule Registry │    │ Context Aggregator   │  │
│  │               │    │                      │  │
│  │ YAML/JSON 配置 │    │ Concur 数据拉取       │  │
│  │ 版本化管理     │    │ 历史报销记录聚合      │  │
│  │ 客户级覆盖     │    │ 出差申请单关联        │  │
│  └───────┬───────┘    │ 员工画像构建          │  │
│          │            └──────────┬────────────┘  │
│          ▼                       ▼               │
│  ┌─────────────────────────────────────────┐    │
│  │          Rule Evaluation Engine          │    │
│  │                                          │    │
│  │  Deterministic Rules  │  AI-Augmented    │    │
│  │  (if-else, 阈值)      │  Rules           │    │
│  │  ├── 金额匹配        │  ├── 场景判断     │    │
│  │  ├── 日期范围        │  │   (是否出差？)  │    │
│  │  ├── 频次统计        │  ├── 凭证-项目    │    │

│  │                      │  └── 异常模式    │    │
│  │                      │      识别        │    │
│  └──────────┬───────────┴──────────────────┘    │
│             ▼                                    │
│  ┌──────────────────────┐                       │
│  │ Decision Aggregator  │                       │
│  │                      │                       │
│  │ 规则结果汇总 + 置信度  │                       │

└────────────────────────────────────────────────┘
```

**规则配置示例（YAML，体现可配置性）：**

```yaml
# rules/client_mccormick.yaml
rules:
  - id: taxi_after_9pm
    name: "非出差晚间打车报销限制"
    type: ai_augmented  # 需要 LLM 判断"是否出差"
    conditions:
      expense_type: "transportation.taxi"
      time_check: "after_21:00"
      trip_status: "no_active_trip"  # 需要关联出差申请单
    action: FAIL
    message_zh: "非出差期间晚9点后打车需符合加班报销政策"
    message_en: "Taxi after 9PM without active trip requires overtime approval"

  - id: phone_number_limit
    name: "电话号码报销数量限制"
    type: deterministic
    conditions:
      expense_type: "communication.phone"
      aggregate: "count_distinct(phone_number, employee_id, fiscal_year)"
      threshold: 2
    action: FAIL
    message_zh: "每位员工每年仅可报销2个电话号码"

  - id: split_billing_detection
    name: "拆单检测"
    type: deterministic
    conditions:
      same_vendor: true
      same_date: true
      count: ">= 2"
      total_amount: "> approval_threshold"
    action: WARN
    message_zh: "检测到同日同供应商多笔报销，合计金额超过审批阈值"
```

### 模块 B：全量凭证核对（替代 5% 抽查）

```
传统模式：
  auditor 抽查金额最大的 5% → 覆盖率低 → 小额欺诈完全漏掉

你们的模式：
  100% 凭证经过 AI 审核 → 分级处理 → 人只看 AI 标出的问题

分级处理逻辑：
  ┌─────────────────────────────────────────────┐
  │ 全量凭证输入（几万到几十万张/月）              │
  │                                               │
  │  Tier 1: 自动通过（~70-80%）                   │
  │  ├── 所有确定性规则通过                        │
  │  ├── OCR 置信度 > 95%                         │
  │  ├── 金额与报销单一致                          │
  │  └── 无异常模式                                │
  │                                               │
  │  Tier 2: AI 标记需关注（~15-25%）              │
  │  ├── 触发至少一条 WARN 规则                    │
  │  ├── OCR 置信度 70-95%（可能识别有误）          │
  │  ├── 金额偏差在容忍范围内                       │
  │  └── → 财务 Dashboard 展示，优先级排序          │
  │                                               │
  │  Tier 3: 自动拒绝（~5%）                       │
  │  ├── 触发 FAIL 规则（发票验真失败/查重命中）    │
  │  ├── 金额严重不匹配                            │
  │  └── → 直接打回给员工                           │
  └─────────────────────────────────────────────┘
```

### 模块 C：海外发票与水单识别（V2 重点）

```
挑战分析：

中国发票（已解决）：
  ├── 版式发票 → 标准 OCR + 税局 API 验真
  ├── 结构化程度高，字段固定
  └── 你们已有成熟方案

海外发票（待解决）：
  ├── 非版式 → 每个国家/商户格式不同
  ├── 语言障碍 → 日文/韩文/阿拉伯文
  ├── 税务差异 → VAT/GST/Sales Tax 规则不同
  └── 关键：不需要"验真"（海外无统一税局接口）
      只需要"信息提取+匹配"

水单（信用卡对账单 + 海外消费小票）：
  ├── 信用卡对账单 → 结构化程度中等，核对金额/日期/商户
  ├── 海外消费小票 → 非标准格式，需要多模态 OCR
  └── 核心需求：水单上的金额/日期 vs 报销单上的金额/日期 是否一致
```

**海外凭证处理方案：**

```
┌─────────────────────────────────────────────┐
│        Overseas Document Processing          │
│                                              │
│  Input: 海外发票/小票/水单 (image/PDF)        │
│                                              │
│  Step 1: 多模态 LLM 提取（Claude Vision）     │
│  ├── 不依赖传统 OCR 模板                      │
│  ├── Prompt: "Extract from this receipt:      │
│  │    amount, currency, date, merchant,       │
│  │    items, tax amount, payment method"      │
│  ├── 支持任意语言（LLM 天然多语言）            │
│  └── 输出: 结构化 JSON                        │
│                                              │
│  Step 2: 币种标准化                           │
│  ├── 自动识别币种                             │
│  ├── 汇率转换（按报销日期汇率）               │
│  └── 与 Concur 中的报销金额核对               │
│                                              │
│  Step 3: 匹配验证                            │
│  ├── 凭证金额 vs 报销金额（允许汇率差异）     │
│  ├── 凭证日期 vs 出差日期范围                 │
│  ├── 商户类型 vs 费用类型                     │
│  └── 信用卡水单金额 vs 发票金额              │
│                                              │
│  ★ 关键 insight：                            │
│  用 Claude Vision 替代传统 OCR 模板化方案      │
│  → 不需要为每个国家维护 OCR 模板              │
│  → 新增国家 = 0 额外开发成本                  │
│  → 这是 AI-native 方案 vs 传统方案的核心差异  │
└─────────────────────────────────────────────┘
```

### 模块 D：数据分析与异常检测（V3 重点）

```
给内审 + CFO 的分析功能：

1. 员工维度分析
   ├── 报销金额趋势（月度/季度同比）
   ├── 异常模式：某员工报销金额突然增长 200%
   ├── 费用类型分布：某员工餐饮占比异常高
   └── 拆单行为检测：同日同供应商多笔

2. 供应商维度分析
   ├── 高频供应商排名
   ├── 异常模式：新注册供应商突然大额交易
   ├── 交叉验证：同一供应商在不同员工处的报价是否一致
   └── 集中度风险：某供应商占某费用类型 80%+

3. 时间维度分析
   ├── 周末/节假日报销异常
   ├── 月底/季度末突击报销
   ├── 报销处理时效分析
   └── 季节性波动 vs 异常波动区分

4. 政策合规分析
   ├── 各部门/地区规则触发率对比
   ├── 高频触发规则排名（帮客户优化政策）
   ├── 规则误报率追踪
   └── 建议：某规则触发率过高但确认率低 → 建议调整
```

### V1 — MVP（4-6 周）：规则引擎 + 全量核对

**目标**：在现有系统上叠加细粒度规则引擎 + 全量 AI 核对

```
架构（叠加在现有系统上）：

现有系统
  ├── 员工上传凭证
  ├── 中国发票 OCR + 验真
  ├── Concur API 双向同步
  └── 财务 Dashboard

V1 新增
  ├── Rule Engine Service（核心新增）
  │   ├── YAML 规则配置（首批 20-30 条）
  │   ├── Context Aggregator（拉取 Concur 出差申请单、历史记录）
  │   ├── Deterministic Rules（金额匹配、频次统计、查重）
  │   └── AI-Augmented Rules（场景判断，调 Claude API）
  │
  ├── Full-Volume Audit Pipeline（核心新增）
  │   ├── 所有凭证过规则引擎（不再抽查）
  │   ├── 三级分流：自动通过 / 需关注 / 自动拒绝
  │   └── 财务 Dashboard 展示分级结果 + 优先级排序
  │
  └── Dashboard Enhancement（小改）
      ├── 规则触发详情展示
      ├── 批量处理界面（财务可以批量通过 Tier 1）
      └── 规则触发统计（哪条规则触发最多）
```

**V1 技术要点（面试展示）：**

```
1. 规则引擎设计模式
   ├── Strategy Pattern: 每条规则是一个 RuleStrategy 实现
   ├── Chain of Responsibility: 规则按优先级链式执行
   ├── 确定性规则先跑（快、便宜、准）
   ├── AI 规则后跑（只对确定性规则无法判断的 case 调 LLM）
   └── 成本优化：~70% 的报销单只过确定性规则就够了，不需要调 LLM

2. Context Aggregator 设计
   ├── 不是每次都实时调 Concur API（太慢）
   ├── 增量同步：每小时拉取 Concur 数据变更
   ├── 本地缓存：员工出差申请单、审批阈值、历史报销
   └── 聚合计算：员工月度累计、供应商频次等预计算

3. 全量核对的性能方案
   ├── 异步队列：报销单提交 → 入队 → 后台处理 → 结果推送
   ├── 批量处理：不是一张一张过规则，而是批量拉取上下文
   ├── 分级处理：Tier 1 自动通过不需要人工介入
   └── 目标：每张凭证 < 5 秒（确定性规则）/ < 30 秒（含 AI 规则）
```

**V1 的价值量化（面试讲故事用）：**

```
Before（客户现状）：
  - 几十条细粒度规则 → 财务手动抽查 → 覆盖率 < 10%
  - 每月几万张凭证 → auditor 仅查金额最大的 5%
  - 发现问题平均延迟：1-3 个月（事后审计）

After（V1 上线后）：
  - 几十条规则 100% 自动执行 → 0 人工 → 实时
  - 全量凭证 AI 核对 → 覆盖率 100%
  - 发现问题延迟：< 1 分钟（提交时即检查）
  - 财务工作量：从逐条审核 → 只看 AI 标出的 15-25%

ROI 估算（以 1000 人企业为例）：
  - 月报销量：~5000 笔
  - 财务审核人力节省：~2 个 FTE × ¥15K/月 = ¥30K/月
  - 欺诈/违规发现率提升：从 5% 覆盖 → 100% 覆盖
  - 你们的收费：SaaS + 凭证量 → 远低于节省的人力成本
```

**V2 面试亮点：为什么用 Claude Vision 而不是传统 OCR？**

```
传统方案（ABBYY/Azure Form Recognizer）：
  ├── 需要为每种发票格式训练模板
  ├── 新增一个国家 = 新增 N 个模板 = 周级开发量
  ├── 非标格式（手写小票、热敏纸）识别率低
  └── 多语言需要分别配置

AI-Native 方案（Claude Vision）：
  ├── Zero-shot 提取：一个 prompt 处理所有国家所有格式
  ├── 新增国家 = 0 开发量（LLM 天然多语言）
  ├── 非标格式识别率高（多模态理解，不是字符匹配）
  ├── 可以理解上下文（"小票上模糊的数字结合菜品推断金额"）
  └── 成本：~$0.01-0.03/张（Claude Sonnet 4.6 vision）

Trade-off（诚实面对）：
  ├── LLM 有幻觉风险 → 置信度 < 80% 时标记需人工复核
  ├── 延迟更高（~3-5 秒 vs 传统 OCR ~0.5 秒）
  └── 成本比传统 OCR 高 → 但省掉了模板维护成本
```

   └── 可视化报告（Dashboard 新 tab）

2. Anomaly Detection Pipeline
   ├── Step 1: 统计基线建立
   │   ├── 员工级别：月均报销金额、费用类型分布
   │   ├── 部门级别：平均值和标准差
   │   └── 公司级别：同比/环比趋势
   │
   ├── Step 2: 异常识别
   │   ├── Z-score > 2 的报销行为
   │   ├── 突变检测（Changepoint Detection）
   │   ├── 聚类异常（Isolation Forest）
   │   └── 关联异常（同供应商跨员工模式）
   │
   └── Step 3: AI 辅助分析
       ├── Claude 对异常模式生成自然语言解读
       ├── 建议调查方向和优先级
       └── 类似案例历史检索

3. Audit Report Generator
   ├── 月度/季度自动生成审计报告
   ├── 包含：异常发现、规则触发统计、趋势分析
   ├── 格式：PDF / 在线交互式报告
   └── 给 CFO / 内审团队 one-click 可用
```

### AI 服务分层设计（关键面试点）

```
┌─────────────────────────────────────────────────┐
│           AI Service Layers                      │
│                                                  │
│  Layer 1: Deterministic（快、准、便宜）            │
│  ├── 金额匹配（精确比较）                         │
│  ├── 日期范围校验                                 │
│  ├── 发票查重（hash 匹配）                        │
│  ├── 频次统计（SQL 查询）                         │
│  └── 成本：$0 / 延迟：< 100ms                    │
│                                                  │
│  Layer 2: OCR + Extraction（中等成本）             │
│  ├── 中国发票：传统 OCR（你们现有方案）             │
│  ├── 海外凭证：Claude Vision（V2）                │
│  └── 成本：$0.01-0.03/张 / 延迟：1-5s            │
│                                                  │
│  Layer 3: AI Reasoning（按需调用）                │
│  ├── 场景判断（"这是出差还是非出差？"）            │
│  ├── 凭证-报销项语义匹配                          │
│  ├── 异常模式识别                                 │
│  └── 成本：$0.01-0.05/次 / 延迟：3-10s           │
│                                                  │
│  ★ 关键设计原则：                                 │
│  能用 Layer 1 解决的不上 Layer 2                   │
│  能用 Layer 2 解决的不上 Layer 3                   │
│  → ~70% 报销单只需要 Layer 1（$0 AI 成本）        │
│  → ~20% 需要 Layer 1+2（$0.01-0.03）             │
│  → ~10% 需要 Layer 1+2+3（$0.02-0.08）           │
│  → 加权平均成本：~$0.01/笔                       │
└─────────────────────────────────────────────────┘
```

### 为什么这个架构设计能打动面试官

```
1. 成本意识
   不是所有东西都用 AI → 分层调用 → 成本可控
   面试话术："我不是AI锤子找钉子，70%的审核用确定性规则解决，
   只在规则搞不定时才调LLM，平均成本¥0.07/笔。"

2. 可扩展性
   规则引擎 YAML 配置 → 新客户新规则 = 配置而非开发
   面试话术："客户的规则半年一变，我们不用改代码，
   实施顾问改YAML配置就行。"

3. 渐进式 AI
   V1: Rules + 偶尔调 AI
   V2: Vision OCR 替代模板 OCR
   V3: ML 异常检测
   面试话术："我的架构让 AI 能力随数据积累自然叠加，
   而不是一开始就over-engineer。"

4. 数据流设计
   每次审核都积累数据 → 自然形成数据飞轮
   immutable audit log → 合规证据链
   面试话术："每处理一笔报销，我的系统同时积累三样东西：

## 五、面试叙事框架

### 30 秒 Elevator Pitch

```
"我们是 SAP Concur 的智能审核增强层。
 Concur 能做基础审批——等级不同限额不同。
 但跨国企业有几十条 Concur 做不了的细粒度规则，
 比如非出差晚9点后打车才可报销、一个员工只能报两个手机号。
 这些规则以前靠财务手动抽查，覆盖率不到10%。

 我们做两件事：
 第一，把这些规则 100% 自动化执行；
 第二，对全量原始凭证做 AI 审核，替代传统5%抽查。
 已经服务味好美、PVH 等跨国零售企业。"
```

### 被问 "为什么不直接在 Concur 里做？"

```
"三个原因：
 1. Concur 的规则引擎是 config-based，不支持语义级判断
    '非出差晚9点后打车' 需要关联出差申请单+时间+费用类型，
    这是 Concur 规则引擎做不了的跨数据源推理。

 2. Concur 不做凭证原件识别
    它只管结构化数据（金额/日期/类型），
    但发票是否真实、水单金额是否匹配，需要 OCR + AI。

 3. 我们是员工的第一入口
    员工先在我们系统完成合规检查，通过后才推到 Concur。
    这个位置让我们能在问题进入审批流之前就拦截。"
```

### 被问 "AI 在这里的角色是什么？不是纯规则引擎就够了吗？"

```
"确定性规则解决 70% 的问题。但剩下 30% 需要 AI：

 1. 场景判断：'这笔打车费是出差还是日常？'
    → 需要理解出差申请单的时间/地点，再判断这笔费用是否在范围内
    → 纯 if-else 写不出来

 2. 海外凭证识别：一张日文的餐厅小票
    → 传统 OCR 需要日文模板，每个国家维护一套
    → Claude Vision 一个 prompt 搞定所有语言和格式

 3. 异常模式：某员工报销行为突然变化
    → 不是违反了哪条规则，而是'不像以前的ta'
    → 需要统计基线 + AI 推理

 所以我的设计是分层的：
 确定性规则优先（快、准、$0）→ 搞不定的才调 AI。
 这也是成本控制的关键：平均每笔报销的 AI 成本 < ¥0.1。"
```

### 被问 "你的技术壁垒是什么？别人也能接 Claude API。"

```
"技术壁垒不在 AI 调用，在三个地方：

 1. 规则知识库
    我们深耕 Concur 多年，积累了跨国企业报销合规的几百条规则模板。
    新客户上线，70% 的规则可以直接用，30% 做客户化配置。
    竞争对手即使技术相同，这个规则库需要几年实施经验才能积累。

 2. Concur 集成深度
    我们的系统是员工报销的第一入口，双向同步 Concur 数据。
    这不是调几个 API 的问题，是理解 Concur 数据模型、
    审批流、费用类型映射的实施 know-how。

 3. 数据飞轮（V3 阶段）
    每处理一笔报销，积累规则触发数据 + 用户反馈 + 行为数据。
    6 个月后我们有行业级的异常检测基线，
    新客户上线 Day 1 就有行业参照，不需要冷启动。"
```

## 六、补充：关键决策日志（展示产品判断力）

| 决策 | 选择 | 放弃的方案 | 为什么 |
|---|---|---|---|
| MVP 范围 | 规则引擎 + 全量核对 | 海外 OCR / 数据分析 | 规则引擎直接解决客户已有痛点，全量核对是最直观的价值提升 |
| 规则存储 | YAML 文件 + 版本管理 | 数据库存储 | 规则半年一变，YAML 更易于 review/diff/rollback |
| AI 调用策略 | 分层（确定性优先） | 全量 AI 审核 | 成本可控 + 确定性规则更可靠 + 速度更快 |
| 海外 OCR | Claude Vision | ABBYY/Azure | Zero-shot 多语言 + 无需维护模板 + 格式无关 |
| 规则配置 | 实施服务配置 | 客户自助 | 规则复杂度高 + 变更频率低 + 已有实施团队 |
| 事前 vs 事后 | 事前为主 + 事后分析 | 纯事后 | 你们是第一入口，事前拦截价值更大 |
