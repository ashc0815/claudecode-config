## 二、你的 V1 技术极简方案（真正 4 周能做完的）

### 技术栈（砍到骨头）

```
总共就 5 个文件的核心逻辑：

1. main.py          — FastAPI，3 个端点
   POST /upload     — 接收凭证图片
   GET  /results    — 查询审计结果
   POST /feedback   — 接收用户反馈（确认/误报）

2. ocr.py           — 调 Mistral OCR API，返回文字
3. audit.py         — 调 Claude Sonnet API，一个 prompt 搞定：
                       输入：OCR 文字
                       输出：结构化 JSON（字段 + 风险标记 + 理由 + 置信度）
4. rules.py         — 10 条 if-else 规则，纯 Python，无依赖
5. notify.py        — 飞书 webhook，发消息卡片（带三个按钮）

基础设施：
  - Supabase（PG + Auth + Storage）— 免费额度够 MVP
  - 部署: Railway 一键部署（$5/月）
  - 前端: 不做。飞书就是前端。上传用飞书群文件 + 飞书表格看结果。
```

### V1 砍掉的东西（重要）

```
❌ Next.js 前端      → 飞书就是界面，不需要独立前端
❌ Redis Queue       → BackgroundTasks 够用（V1 月处理量 < 5000 笔）
❌ 多 Agent 编排      → 一条管线：OCR → Claude → Rules → 飞书通知
❌ 多 OCR 引擎投票    → 只用 Mistral OCR，不够准让用户手动纠正
❌ 评测 CI Pipeline  → 手动跑 eval 脚本就行，别搭 CI
❌ 多币种             → 只支持人民币
❌ 用户管理/权限      → Supabase Auth 自带，不自建
❌ 数据可视化 Dashboard → 飞书多维表格 = 免费 Dashboard
```

### V1 的真实开发量

```
Week 1: OCR + Claude prompt 调通（2-3 天），10 条规则写完（1 天）
Week 2: FastAPI 服务 + Supabase 集成 + 飞书通知（3 天）
Week 3: 评测集 100 张 + 调 prompt 到 Precision ≥ 70%（全周）
Week 4: 找 3 个朋友的公司测试 + 修 bug（全周）

总代码量预估: < 1500 行 Python
总依赖: fastapi, httpx, supabase-py, 就这仨
总成本: Supabase 免费 + Railway $5/月 + Claude API ~$30/月（1000 笔） = $35/月
```

## 三、你的叙事天花板（面试/融资时讲的故事）

### 叙事层 1：问题有多大

```
"全球有 4 亿家 SME，其中 85% 没有专职内审。
 每年因内部欺诈和财务错误造成的损失占营收的 5%（ACFE 2024 报告）。
 一家年营收 ¥2000 万的 SME，每年可能因此损失 ¥100 万——
 但他们请不起 MindBridge（$50K/年）或四大审计师（$500/小时）。

 这是一个 $100B+ 的全球市场，但现有方案只服务 Top 0.1% 的企业。
 剩下的 99.9% 怎么办？"
```

### 叙事层 2：为什么现在

```
"三个变量同时成熟了：

 1. LLM 成本暴降
    Claude Sonnet 处理一张发票的成本 = ¥0.15
    → 1000 笔/月的审计成本 = ¥150 = 一顿火锅
    → 两年前这个成本是 ¥15/张，贵 100 倍，SME 用不起

 2. 多模态 OCR 能力突破
    Mistral OCR + Claude Vision 可以直接读手写收据、模糊扫描件
    → 两年前需要训练专用 OCR 模型，成本 ¥50 万+

 3. 企业 IM 成为工作入口
    飞书/钉钉/企微日活数亿
    → 审计结果直接推送到 CFO 已经在用的工具里
    → 不需要教用户学新系统 → 获客成本极低"
```

## 四、技术极简 ↔ 叙事天花板的「桥梁设计」

关键问题：面试官/投资人会问 **"你现在做的这个小东西，跟你讲的大故事之间有什么关系？"**

答案是 **架构预留 + 数据飞轮**：

```
V1 虽然只有一条管线，但做了 3 个"埋点"：

1. 结构化输出 Schema
   Claude 返回的 JSON 有固定 schema:
   {
     "vendor_name": "...",
     "amount": 49999,
     "currency": "CNY",
     "date": "2026-03-12",
     "risk_flags": [
       {"rule": "threshold_proximity", "confidence": 0.82, "detail": "..."},
       {"rule": "weekend_invoice", "confidence": 0.95, "detail": "..."}
     ],
     "ai_reasoning": "...",
     "citations": ["原始凭证第2行", "金额字段"]
   }

   为什么重要：
   → V2 加自定义规则时，直接在 risk_flags 数组里追加
   → V3 做供应商画像时，vendor_name 已经被标准化提取了
   → V4 做信用评分时，所有历史 JSON 都可以聚合分析
   → 如果 V1 没有固定 schema，后面每次升级都要迁移数据

2. 反馈数据表（feedback table）
   V1 只用来收「确认/误报」
   → V2 变成模型微调的训练数据
   → V3 变成行业基线的统计来源

3. 审计日志（immutable audit_log table）
   V1 只是 "谁在什么时候上传了什么"
   → V2 变成合规证据链（"我们的 AI 在 X 时间标记了 Y 风险，Z 人在 W 时间做了处理"）
   → V3 这就是金融机构愿意付费的东西：可审计的决策记录
```

### 面试时的一句话总结

```
"我的 V1 只有 1500 行 Python 和一个飞书机器人。
 但我设计了三个数据结构——结构化风险标记、用户反馈、审计日志——
 这三个表让产品从'凭证扫描工具'长成'财务合规平台'的路径
 是自然演进，不是推倒重来。

 这就是为什么 V1 做得极简但天花板极高：
 不是因为我画了一个大饼，
 而是因为数据结构的设计让每个后续版本都是在现有基础上叠加，
 而不是重写。"
```

## 五、面试时的「技术极简 × 叙事天花板」话术模板

### 被问 "你现在有什么？"

```
诚实但有力：

"一个 1500 行的 Python 服务，在 3 家 SME 跑了 2 周。
 处理了 800+ 张凭证，发现了 12 笔已确认的异常——
 其中一笔是 ¥8 万的虚增报价，客户在追回中。

 技术上就是 OCR + Claude + 10 条规则 + 飞书通知。
 没有花哨的架构，没有多 Agent，没有 ML。
 但它 work。"
```

### 被问 "这怎么做大？"

```
用数据飞轮讲：

"每处理一张凭证，我的系统积累三样东西：
 1. 一个结构化的风险标记（供应商 + 金额 + 风险类型）
 2. 一条用户反馈（确认 or 误报）
 3. 一条审计日志

 100 个客户跑 6 个月，我就有：
 - 60 万笔交易的风险数据库
 - 行业级别的异常基线（制造业 vs 贸易 vs 服务业的 pattern 完全不同）
 - 供应商信用图谱（同一个供应商在不同客户那里的表现交叉验证）

 这些数据让我从 Tool 变成 Platform：
 - 新客户上线时，直接用行业基线，不需要冷启动
 - 供应商在 A 客户那里被标红，B 客户也会收到预警
 - 金融机构可以基于我的数据做 SME 信用评估

 V1 的 1500 行代码不是终点，是数据收集器的起点。"
```

### 被问 "竞品怎么办？MindBridge 融了 $4000 万"

```
用定位差异讲：

"MindBridge 做的是 enterprise audit platform——
 他们的客户是 EY、PwC 这种四大级别的审计师事务所。
 $50K/年的定价说明他们不打算服务 SME。

 这就像 Salesforce vs HubSpot：
 Salesforce 做 Fortune 500，HubSpot 做 SMB。
 HubSpot 市值 $300 亿，证明了 SMB 市场足够大。

 我做的是审计领域的 HubSpot：
 - 不需要实施团队，飞书群里直接用
 - 不需要 $50K 预算，¥499/月
 - 不需要审计专业知识，AI 帮你看

 MindBridge 不会降价来抢我的客户——
 就像 Salesforce 不会做 $50/月的 CRM。"
```

### 被问 "为什么用 Claude 不用 GPT？"

```
用产品判断讲，不用技术术语：

"三个原因：
 1. Claude 的结构化输出在金融场景的准确率最高
    （Vals AI Finance Agent Benchmark，Claude 排名第一）
 2. Anthropic 专门做了 Claude for Financial Services，
    有 LSEG/FactSet/S&P 的数据接入——
    虽然 V1 用不到，但 V3 做供应商尽调时直接接入
 3. Anthropic 的数据政策明确：客户数据不用于模型训练——
    金融客户签约时这一条直接省掉两周的法务审核

 但更重要的是：我的架构不绑死 Claude。
 audit.py 里调 Claude API 的代码只有 30 行。
 如果明天 GPT-5 更好更便宜，我换一个 import 就行。
 核心资产是 prompt、规则、数据——不是哪个模型。"
```
